# 텍스트+첨부파일 기반 자동등록 (Smart Auto Register)

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 회계책임자가 지출 내역을 등록하려면 계정/과목/지출유형 3단계를 직접 선택해야 하고, 보전 여부도 별도로 확인해야 함. 현장에서 영수증 받고 "명함 인쇄 50만원" 같은 메모만 남기면 나중에 등록할 때 어떤 유형인지 다시 고민해야 함 |
| **Solution** | 내용 텍스트(예: "명함 인쇄 50만원 양지디자인")와 첨부파일(영수증 사진)을 입력하면, 시스템이 지출유형 자동매핑 + 코드값 자동설정 + 보전여부 자동판별 + OCR 보조정보 추출을 한 번에 처리하여 등록 |
| **Function UX Effect** | 메모 한 줄 + 사진 한 장 → 계정/과목/지출유형/보전여부 전부 자동 → 확인 후 1클릭 등록. 현장 메모 수준의 입력으로 정확한 회계 장부 생성 |
| **Core Value** | 회계 등록의 최종 진입장벽 제거. "메모장에 적어놓고 나중에 입력"하던 패턴을 "그 자리에서 바로 등록"으로 전환 |

## 1. 배경 및 문제

### 현재 상황
- **마법사**: 카드 선택 → 폼 입력 → 확인 (3단계, 초보자 친화적이지만 여전히 수동 선택)
- **영수증 OCR**: 파일 업로드 → AI가 날짜/금액/거래처 추출 → 사용자가 계정/과목/지출유형 직접 선택
- **보전 판별**: Step 3에서 지출유형 선택 후 뱃지로 표시 (자동이지만 등록 프로세스와 분리)

### 사용자의 실제 행동 패턴
1. 선거 현장에서 영수증을 받음 (현수막 제작비 30만원)
2. 핸드폰 카메라로 영수증 촬영
3. "현수막 제작 30만원 OO간판점" 메모
4. **나중에** 회계 프로그램에서 계정/과목/지출유형을 하나하나 선택하며 등록
5. 보전 대상인지 별도로 확인

→ 3~5번 과정을 자동화하여 2번 직후에 바로 등록할 수 있게 하는 것이 목표

### 대상 사용자
- 선거 현장에서 실시간으로 지출을 기록하는 회계책임자
- 메모 수준의 입력만으로 정확한 장부를 만들고 싶은 사람
- 보전 여부를 매번 확인하기 어려운 초보자

## 2. 목표

1. **텍스트 한 줄 + 파일 한 장으로 자동 등록** (수동 코드 선택 0회)
2. **키워드 기반 지출유형 자동 매핑** (기존 `wizard-mappings.ts` + `expense-types.ts` 활용)
3. **보전 여부 즉시 표시** (기존 `getReimbursementStatus()` 활용)
4. **OCR 보조**: 첨부파일이 있으면 Gemini Vision으로 금액/날짜/거래처 교차 검증
5. **기존 시스템 100% 호환**: 기존 acc_book 테이블에 동일하게 저장

### 3.2 텍스트 분석 로직

입력 텍스트에서 다음 정보를 추출:

```text
"명함 인쇄 50만원 양지디자인 4월10일 카드"
  │      │      │        │       │
  │      │      │        │       └─ 결제수단 매칭
  │      │      │        └─ 날짜 패턴 매칭
  │      │      └─ 거래처명 (숫자/키워드 아닌 나머지)
  │      └─ 금액 패턴 매칭 (50만원, 500000, 500,000)
  └─ 키워드 → 지출유형 매핑 (wizard-mappings.ts keywords)
```

**텍스트 파싱 규칙:**
1. **금액**: `\d+만원`, `\d{1,3}(,\d{3})*원?`, `\d+` 패턴
2. **날짜**: `\d{1,2}월\d{1,2}일`, `\d{4}[-/.]\d{1,2}[-/.]\d{1,2}`, `어제`, `오늘`, `그제`
3. **결제수단**: "카드", "현금", "계좌", "이체", "수표" 키워드
4. **지출유형**: `wizard-mappings.ts`의 keywords 배열과 매칭 (기존 searchWizardTypes 로직 확장)
5. **거래처**: 금액/날짜/결제수단/키워드를 제외한 나머지 텍스트

### 3.3 키워드 → 지출유형 매핑 확장

기존 `searchWizardTypes()`는 카드 필터링용. 새로운 함수 필요:

```typescript
/** 텍스트에서 가장 적합한 WizardType + 하위 지출유형을 추론 */
function inferExpenseType(
  text: string,
  types: WizardType[]
): {
  wizardType: WizardType | null;       // 매칭된 카드
  expGroup1: string;                    // 지출유형 대분류
  expGroup2: string;                    // 지출유형 중분류
  expGroup3: string;                    // 지출유형 소분류
  confidence: number;                   // 매칭 신뢰도 (0~1)
}
```

**매칭 우선순위:**
1. 정확한 level2/level3 키워드 매칭 (신뢰도 0.9)
2. WizardType keywords 매칭 (신뢰도 0.7)
3. 부분 문자열 매칭 (신뢰도 0.5)
4. 매칭 없음 → "기타" 카드 (신뢰도 0.0)

### 3.4 OCR 보조 (첨부파일 있을 때)

기존 `/api/receipt-scan` API 재사용:
- 텍스트 입력값과 OCR 추출값을 비교
- 일치하면 ✓, 불일치하면 ⚠ 표시
- OCR 값이 더 정확해 보이면 자동 대체 제안

### 3.5 보전 여부 자동 표시

기존 `getReimbursementStatus()` 함수 활용:
- 지출유형이 결정되면 즉시 보전 여부 뱃지 표시
- 보전/미보전/선거비용외/판별불가 4단계

## 4. 구현 범위

### 구현 대상
- `lib/text-parser.ts` — 텍스트 파싱 로직 (금액/날짜/결제수단/거래처 추출)
- `dashboard/wizard/page.tsx` — "빠른 등록" 탭 추가 (기존 마법사 페이지에 통합)
- `lib/wizard-mappings.ts` — inferExpenseType 함수 추가

### 재사용 코드
- `wizard-mappings.ts` — WizardType 데이터 + resolveCodeValues
- `expense-types.ts` — getReimbursementStatus (보전 판별)
- `/api/receipt-scan` — Gemini Vision OCR (첨부파일 분석)
- `/api/acc-book` — 저장
- `/api/evidence-file` — 증빙파일 업로드
- `useCodeValues()` — 계정/과목 코드 조회
- `CustomerSearchDialog` — 거래처 검색/등록

### 구현 제외
- AI(LLM) 기반 텍스트 분석 (키워드 매칭으로 충분, Gemini API 호출 최소화)
- 음성 입력 (향후 고려)
- 일괄 텍스트 입력 (여러 건 한 번에)
- 수입 자동등록 (1차는 지출만, 수입은 구조가 단순하여 마법사로 충분)

### 텍스트 파서 설계

```typescript
// lib/text-parser.ts
interface ParsedExpenseText {
  amount: number | null;           // 추출된 금액
  date: string | null;             // 추출된 날짜 (YYYY-MM-DD)
  payMethod: string | null;        // 결제수단 코드 ("118", "119" 등)
  customerName: string | null;     // 추출된 거래처명
  content: string;                 // 정리된 내역 텍스트
  keywords: string[];              // 지출유형 매칭용 키워드
}

function parseExpenseText(text: string): ParsedExpenseText
```

### 데이터 흐름

```text
텍스트 입력 → parseExpenseText()
  → 키워드 추출 → inferExpenseType() → WizardType + 지출유형 3단계
  → resolveCodeValues() → 계정/과목 자동 설정
  → getReimbursementStatus() → 보전 여부 판별

(첨부파일 있으면 병렬로)
  → /api/receipt-scan → OCR 결과
  → 텍스트 파싱 결과와 비교/보완
  
→ 사용자 확인 화면 → 수정 가능 → 등록하기
  → /api/acc-book insert + /api/evidence-file
```

## 8. 성공 지표

- 텍스트 입력 → 자동 매핑 정확도 **80% 이상** (키워드 매칭 기반)
- 금액/날짜 추출 정확도 **95% 이상** (정규식 기반)
- 1건 등록 소요 시간 **30초 이내** (기존 마법사 1분 대비 50% 단축)
- OCR 보조 시 데이터 정확도 **향상** (텍스트 단독 대비)
