# Plan: 선거회계처리 RAG 구축 및 채팅 UI 구현

> **Feature**: rag-chat
> **Created**: 2026-03-25
> **Status**: Plan

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 정치자금 회계처리 시 법률/규칙/서식 관련 질문에 즉시 답변을 받기 어려움. 76p 도움말, 57p 매뉴얼, 73p 가이드북 등 방대한 참고자료를 매번 찾아봐야 함 |
| **Solution** | 선관위 공식 문서(도움말, 매뉴얼, 가이드북, 법률)를 벡터DB에 임베딩하고, 대시보드 내 채팅 UI를 통해 자연어로 질의응답할 수 있는 RAG 시스템 구축 |
| **Function UX Effect** | 대시보드 우하단 채팅 버블 → 클릭 시 채팅 패널 열림 → "후원금 한도가 얼마인가요?" 같은 질문에 근거 문서 인용과 함께 즉시 답변 |
| **Core Value** | 회계 실무자가 법률 전문가 없이도 정확한 회계처리 가능. 오류 감소 + 업무 시간 50% 이상 단축 |

## 1. 배경 및 목적

### 1.1 현재 문제
- 정치자금법, 정치자금사무관리규칙 등 복잡한 법률 체계
- 회계처리 시 매번 도움말/매뉴얼을 검색해야 하는 비효율
- 서식 작성법, 한도액 계산, 지출유형 분류 등 반복되는 질문
- 비전문가(초선 후보자/후원회)가 회계처리 시 오류 발생 빈도 높음

### 1.2 목표
1. 선관위 공식 문서를 기반으로 한 정확한 회계 안내 시스템
2. 대시보드 내 상시 접근 가능한 채팅 인터페이스
3. 답변 시 근거 문서(출처)를 함께 표시하여 신뢰성 확보
4. 현재 작업 맥락(수입 등록 중, 지출 등록 중 등)에 따른 맥락 인식 답변

## 2. RAG 시스템 설계

### 2.1 데이터 소스 (임베딩 대상)

| # | 문서명 | 페이지 | 내용 |
|---|--------|--------|------|
| 1 | 정치자금회계관리프로그램 도움말.pdf | 76p | 프로그램 사용법, 메뉴별 설명 |
| 2 | 사용자매뉴얼 (예비)후보자 및 후원회용.pdf | 57p | 회계실무 절차, 서식 작성 예시 |
| 3 | 후원회설립가이드북.pdf | 73p | 후원회 설립/운영/해산 절차 |
| 4 | PROGRAM_DESIGN.md | ~4000줄 | 시스템 설계서, 코드체계, 비즈니스 로직 |
| 5 | 정치자금법 주요 조문 | - | 법률 텍스트 (제34조~제49조 등 핵심 조문) |
| 6 | 서식 템플릿 설명 (FORM_TEMPLATES.md) | - | 45종 서식 설명 및 작성 가이드 |

### 2.2 벡터 DB 및 임베딩

| 구분 | 선택 | 이유 |
|------|------|------|
| 벡터DB | **Supabase pgvector** | 기존 Supabase 인프라 활용, 별도 서비스 불필요 |
| 임베딩 모델 | **OpenAI text-embedding-3-small** | 한국어 성능 우수, 1536차원, 비용 효율적 |
| LLM | **Claude 3.5 Sonnet (Anthropic API)** | 한국어 품질 최고, 법률 텍스트 이해도 높음 |
| 청크 크기 | 500~800자 (overlap 100자) | 법률 문서 특성상 문단 단위 |

### 2.3 임베딩 파이프라인

```
PDF/MD 문서 → 텍스트 추출 → 청크 분할 → 임베딩 생성 → Supabase pgvector 저장

테이블: rag_documents
  - id: BIGINT (PK)
  - content: TEXT (원본 텍스트 청크)
  - embedding: VECTOR(1536)
  - metadata: JSONB (source, page, section, category)
  - created_at: TIMESTAMPTZ

인덱스: ivfflat (lists=100) on embedding
```

### 2.4 검색 + 생성 흐름

```
사용자 질문
  ↓
질문 임베딩 생성 (text-embedding-3-small)
  ↓
pgvector 유사도 검색 (cosine similarity, top-5)
  ↓
검색 결과 + 시스템 프롬프트 + 질문 → Claude API
  ↓
답변 생성 (근거 문서 인용 포함)
  ↓
채팅 UI에 표시 (답변 + 출처 표시)
```

## 3. 채팅 UI 설계

### 3.1 UI 컴포넌트

```
┌─────────────────────────────────────┐
│  대시보드 (기존)                      │
│                                     │
│                                     │
│                          ┌────────┐ │
│                          │ 💬     │ │  ← 채팅 버블 (우하단 고정)
│                          └────────┘ │
└─────────────────────────────────────┘

클릭 시:
┌─────────────────────────────────────┐
│  대시보드              ┌──────────┐ │
│                        │ 채팅 패널 │ │
│                        │          │ │
│                        │ Q: ...   │ │
│                        │ A: ...   │ │
│                        │          │ │
│                        │ [입력란] │ │
│                        └──────────┘ │
└─────────────────────────────────────┘
```

### 3.2 기능 요구사항

| # | 기능 | 설명 |
|---|------|------|
| F1 | 채팅 버블 | 우하단 고정, 미읽은 메시지 뱃지 |
| F2 | 채팅 패널 | 슬라이드 인/아웃, 드래그 리사이즈 |
| F3 | 메시지 표시 | 마크다운 렌더링, 코드 블록, 테이블 |
| F4 | 출처 표시 | 답변 하단에 참고 문서명 + 페이지 번호 |
| F5 | 맥락 인식 | 현재 페이지(수입/지출/재산 등)에 따른 맥락 전달 |
| F6 | 자주 묻는 질문 | 빠른 접근 버튼 (한도액, 서식 작성법 등) |
| F7 | 대화 이력 | 세션 내 대화 유지, 새로고침 시 초기화 |
| F8 | 스트리밍 응답 | 답변이 실시간으로 타이핑되듯 표시 |

## 4. API 설계

### 4.1 새로운 API 라우트

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/chat` | RAG 채팅 (질문 → 답변) |
| POST | `/api/rag/embed` | 문서 임베딩 (관리자 전용) |
| GET | `/api/rag/search` | 벡터 유사도 검색 (디버깅용) |

### 4.2 POST /api/chat 스펙

```json
// Request
{
  "message": "후원금 한도액이 얼마인가요?",
  "context": {
    "currentPage": "/dashboard/income",
    "orgType": "supporter"
  }
}

// Response (streaming: text/event-stream)
data: {"type": "text", "content": "후원금 한도액은 "}
data: {"type": "text", "content": "후원회 유형에 따라 ..."}
data: {"type": "sources", "sources": [
  {"title": "도움말", "page": 23, "snippet": "..."},
  {"title": "정치자금법 제12조", "snippet": "..."}
]}
data: [DONE]
```

## 5. 기술 스택

| 구분 | 기술 | 이유 |
|------|------|------|
| 벡터DB | Supabase pgvector | 기존 인프라 활용 |
| 임베딩 | OpenAI API (text-embedding-3-small) | 한국어 성능, 비용 효율 |

| PDF 파싱 | pdf-parse (서버사이드) | PDF 텍스트 추출 |
| 스트리밍 | Server-Sent Events (SSE) | Next.js API Route 호환 |
| 마크다운 | react-markdown + remark-gfm | 채팅 답변 렌더링 |

## 7. 구현 우선순위

### Phase 1: 벡터DB + 임베딩 파이프라인
1. Supabase에 pgvector 확장 활성화 + rag_documents 테이블 생성
2. PDF 텍스트 추출 스크립트 (pdf-parse)
3. 청크 분할 + OpenAI 임베딩 생성 + Supabase 저장
4. 벡터 유사도 검색 함수 (match_documents RPC)

### Phase 2: Chat API
5. `/api/chat` 라우트 (SSE 스트리밍)
6. 시스템 프롬프트 설계 (한국어 정치자금 전문가 역할)
7. 검색 결과 + 질문 → Claude API 호출
8. 출처 정보 포함한 응답 생성

### Phase 3: 채팅 UI
9. ChatBubble 컴포넌트 (우하단 고정)
10. ChatPanel 컴포넌트 (메시지 목록 + 입력란)
11. 스트리밍 응답 표시 (SSE 파싱)
12. 마크다운 렌더링 + 출처 표시
13. 자주 묻는 질문 Quick Actions

### Phase 4: 고도화
14. 맥락 인식 (현재 페이지 기반 시스템 프롬프트 조정)
15. 대화 이력 관리 (멀티턴 대화)
16. 임베딩 갱신 스크립트 (문서 변경 시)

## 9. 리스크 및 고려사항

| 리스크 | 대응 |
|--------|------|
| 법률 해석 오류 | 반드시 출처 표시, "법률 전문가 확인 권장" 면책 문구 |
| 임베딩 품질 | 청크 크기 조정, metadata 필터링으로 정확도 향상 |
| API 비용 | 캐싱 (동일 질문 재사용), rate limiting |
| 응답 속도 | 스트리밍으로 체감 속도 개선 |
| 한국어 토큰 효율 | Claude의 한국어 토큰 효율이 GPT보다 우수 |
